## Problem

Tapping Play on iPad showed the playing state but produced no sound, while it worked on desktop.

## Root cause

iOS/iPadOS Safari starts Web Audio in the **"ambient" audio session**, which the hardware mute switch (and Control Center mute) silences, even though native apps still play. `Tone.start()` resumes the suspended `AudioContext` but never promotes the tab to the **"playback"** session, so the transport ran and the UI flipped to "playing" while output stayed muted.

## Fix

New `src/utils/audio-unlock.ts` exposes `unlockAudio()`, called on the first user gesture:

1. `await Tone.start()` — resumes the context (all browsers).
2. Plays a tiny silent `<audio>` element (`playsinline`, near-zero volume) to flip iOS into the **playback** audio session that ignores the mute switch.
3. Primes a 1-sample Web Audio buffer for older iOS that needs a rendered node.

Idempotent (only the first call does work). Wired into `play()` and `playNote()` in `usePlayback.ts`, replacing the bare `Tone.start()` calls.
