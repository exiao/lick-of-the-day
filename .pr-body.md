## Why it was slow

Every "New Lick" click made a **blocking call to Claude Sonnet** that emitted a large note-by-note JSON blob. Measured 10-15s per 4-bar lick. Nothing broken, just a full LLM round-trip per click with no caching and no prefetch.

## Haiku vs Sonnet (measured, same prompt, 4 bars)

| Genre | Sonnet | Haiku 4.5 |
|---|---|---|
| jazz | 13.2s | 6.2s |
| blues | 10.4s | 5.9s |
| funk | 15.1s | 7.9s |

**~2x faster.** Quality (read the actual note data): jazz comparable, blues slightly richer on Sonnet, **funk actually better on Haiku** (it followed the "tight 16th-note" brief; Sonnet only used 8ths/quarters). 100% JSON parse success on both. Verdict: ship Haiku.

## Changes

**Backend** (shared `lick-config.ts`, applied to all 5 call sites: Vercel `api/random` + `api/daily`, Cloudflare `functions/api/random` + `daily`, cron worker):
- `claude-sonnet-4-6` → `claude-haiku-4-5`
- Ephemeral **prompt caching** (`cache_control`) on the 19KB system prompt — cuts input latency/cost after the first call. (Cache write confirmed registering in testing; hits will land in prod since it calls Anthropic directly.)
- `max_tokens` 2048 → 1536 (a 4-bar lick is ~1200 output tokens)

**Frontend:**
- **Prefetch** the next random lick in the background, so "New Lick" is instant: swap in the prefetched one, then queue the next. Warms on mount after the daily lick loads.
- **Random genre**, and **removed the genre + bars selectors**. Prefetch needs a known genre ahead of time, and a manual selector fights the instant-swap UX. Keeps it a fun "what do I get" surprise.
